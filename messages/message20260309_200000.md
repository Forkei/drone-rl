## clip_fraction analysis: the smoking gun

```
Step range          clip_fraction   DR level        Notes
0 (first update)    0.586           L0-clean        Loaded from v1 best_model
25k-75k             0.578-0.59      L1-mild
75k-150k            0.590-0.593     L2-moderate
150k-300k           0.592-0.598     L3-strong       Warmup was designed to hold here if >0.4
300k-1500k          0.692-0.714     L4-full         Clip fraction rose further at full DR
```

**Key finding: clip_fraction=0.59 appeared at step 0 on CLEAN physics.**

This proves the value function mismatch is NOT caused by the DR dynamics change.
It was already present in the model before any training resumed. v1 best_model
cannot be fine-tuned by PPO from this checkpoint regardless of:
- Reward changes (v2, v2b: same pattern)
- Dynamics changes (DR resume: same)
- Gradual dynamics introduction (DR warmup: same)

The adaptive hold never triggered because clip_fraction reads as nan between PPO
update calls. The `not np.isnan(clip_frac) and clip_frac > 0.4` guard evaluates
False when clip_frac=nan → no hold → ramp advanced as scheduled.
The actual policy-update clip_fraction was 0.59+ throughout.

## Why clip_fraction=0.59 from step 0?

With clip_range=0.1, clip_fraction=0.59 means 59% of policy updates are clipped.
This indicates the policy gradient direction (from current rollouts) is highly
inconsistent with the policy from the previous PPO epoch. Multiple hypotheses:

1. **Training reached saturation**: v1 best_model is at a local optimum.
   Any gradient step from new rollouts exceeds clip_range=0.1 because the
   advantage landscape has negligible gradient near the optimum.
   At a true optimum, policy gradient loss ≈ 0 → advantage ≈ 0 →
   policy ratio stays near 1 → clip_fraction should be LOW.
   Contradiction: clip_fraction is HIGH. So not a true optimum.

2. **High entropy policy**: std=1.15 (very high) → wide action distributions
   → large variance in sampled actions → large gradient steps needed per epoch
   → repeated re-sampling clips more updates. This is likely the primary cause.

3. **Value function is miscalibrated**: explained_variance starts at 0.881 (OK)
   but drops to 0.62-0.73 rapidly. Returns are highly variable due to
   non-deterministic episodes. V(s) cannot track rapidly → large A(s,a)
   → large policy gradient → clipping.

## What this means for v1 best_model

v1 best_model achieved 2/9 on clean NavAviary. The policy is high-entropy (std=1.15),
meaning it has learned approximate navigation but not precision settling.
It cannot be improved via standard PPO fine-tuning because:
- The advantage estimates are too noisy (high std)
- PPO clips itself into stagnation on any continued training

Every fine-tuning attempt confirmed this: eval reward oscillates between -1.89
and +1.14 (vs v1's 12.58), never recovers, 0/9 in every attempt.

## Options forward

**Option A: Value head reset** (targeted fix for the diagnosed problem)
Reset only the value network weights (`model.policy.mlp_extractor.value_net`)
while keeping the policy network frozen. Train at lr=1e-5 for 50k steps (warm
the value head only), then unfreeze policy at lr=3e-5.
This directly addresses the miscalibrated V(s) without changing the policy.
Risk: value head may not fully recalibrate within budget.
ETA: ~30 min.

**Option B: Policy distillation** (sidestep fine-tuning entirely)
Generate expert demonstrations from v1 best_model on clean NavAviary.
Train a NEW policy via behavioral cloning (supervised learning) on these demos.
BC doesn't suffer from PPO's clipping instability.
Then fine-tune the BC-initialized policy with PPO from scratch.
This preserves the navigation skill while allowing clean retraining.
Risk: BC may not perfectly reproduce the approach behavior.
ETA: ~1-2 hours (demo collection + BC training + fine-tune).

**Option C: Accept 2/9 as baseline, change objectives**
v1 best_model (2/9) is the current best clean-physics policy.
Neither DR nor settling improvements worked within 5 attempts.
Consider changing problem scope: train for higher success rate at fixed 0.3m range
(near_0.3m case), rather than 2.0m range generalization.
This would be a fundamentally different training objective.

**Option D: Entropy annealing + PPO restart**
Train a new policy from scratch with explicit entropy decay schedule:
entropy_coef anneals from 0.05 → 0.001 over 1M steps.
This forces the policy to converge to a lower-entropy (more precise) behavior.
May address the settling problem naturally.
Start from v1 best_model but with explicit entropy constraint via PPO penalty.
ETA: ~1.5 hours.

## My lean: Option A first (30 min test), then Option B if A fails.

Value head reset is the minimal targeted intervention. If the diagnosis is correct
(V(s) miscalibration), it should work. If clip_fraction persists after reset,
then the problem is the policy network itself, and Option B (distillation) becomes
the right path.
