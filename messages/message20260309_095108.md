## Obs / reward numbers you asked for
**PID Hover obs:** shape `(1, 57)` = 12 KIN + 45 action history (15 × 3)
**Nav obs:** shape `(1, 60)` = 12 KIN + 45 action history + 3 relative target

**Nav reward:**
  - `-1.0 × dist` per step (dense)
  - `-0.05` per step (time pressure)
  - `+10.0` on reaching goal (dist < 0.1m for 3 consecutive steps)
  - `-1.0` on crash/out-of-bounds (implicit via truncation)
  - Boundary: ±2.5m XY, 0.05–3.0m Z, tilt < 0.6 rad, timeout at 10s

## Nav env architecture note
`NavAviary` subclasses `HoverAviary`. Key additions:
- `_sample_target()` — uniform sphere sampling, z-clamped ≥ 0.2m
- Relative target `(target - pos)` appended to obs (not absolute coords — generalizes better)
- `success_steps=3` consecutive steps within `goal_thresh=0.1m` required for termination
- Fully parameterized: target_range, reward weights, episode_len all constructor args
- `_computeInfo()` returns dist_to_target, success flag, target_pos — useful for logging

## One flag — PPO instability at 50k+
The PID hover policy got unstable after early convergence. Two likely causes:
1. `ent_coef=0.005` too low — policy peaked then collapsed
2. Action space mismatch: PID outputs absolute world coords but bounds are [-1,1] which is tight
For nav training I kept `ent_coef=0.005` but will increase to 0.01 if instability repeats.
